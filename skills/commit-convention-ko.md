# 한국어 커밋 컨벤션

## 목적
- 커밋 메시지를 한국어 중심으로 통일한다.
- 변경 의도를 빠르게 파악할 수 있게 유지한다.

## 기본 형식
- `<type> : <한국어 제목>`

예시:
- `feat : 어드민 마이페이지 기본 진입 경로 추가`
- `fix : Cloudflare Pages 동적 라우트 edge 런타임 누락 수정`
- `refactor : admin API 호출 모듈을 도메인별로 분리`

## type 목록
- `feat` : 기능 추가
- `fix` : 버그 수정
- `refactor` : 동작 변화 없는 구조 개선
- `docs` : 문서 수정
- `test` : 테스트 추가/수정
- `chore` : 빌드/설정/의존성 등 기타 작업
- `style` : 코드 포맷/스타일 변경
- `build` : 빌드 시스템 변경
- `ci` : CI 설정 변경
- `revert` : 이전 커밋 되돌리기

## 작성 규칙
- 제목은 한국어로 작성한다.
- 제목은 한 줄로 간결하게 작성한다.
- 가능하면 72자 이내로 유지한다.
- 무엇을 했는지보다 왜 필요한지 드러나게 작성한다.

## 의도 단위 분리 커밋 규칙
- 사용자가 `커밋해줘`라고만 말해도 변경 의도별로 자동 분리 커밋한다.
- 서로 다른 목적(기능 추가, 버그 수정, 리팩터링, 문서/설정)은 한 커밋에 섞지 않는다.
- 커밋 경계는 `왜 같이 롤백되어야 하는가` 기준으로 잡는다.
- 생성물(dist, lockfile)은 해당 의도 커밋에만 포함하고, 무관한 생성물은 분리한다.

### 분리 기준
- `feat`와 `fix`는 반드시 분리한다.
- `refactor`는 동작 변화가 있으면 `fix` 또는 `feat`로 분리한다.
- `docs`/`chore`는 런타임 동작 변경 커밋과 분리한다.

### 기본 처리 순서 ("커밋해줘" 요청 시)
1. 변경 파일을 의도별로 그룹화한다.
2. 각 그룹마다 lint/build/test를 필요한 범위로 검증한다.
3. 그룹별로 한국어 커밋 메시지(`<type> : <제목>`)로 순차 커밋한다.
4. 필요 시 마지막에 생성물 전용 커밋을 별도로 만든다.

## 권장 체크리스트
- 변경 범위가 하나의 의도로 묶였는가?
- 관련 lint/build/test가 통과했는가?
- 커밋 제목만 봐도 목적이 이해되는가?
